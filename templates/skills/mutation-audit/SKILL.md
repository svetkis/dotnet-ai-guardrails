# Mutation Audit — Skill

## Context Marker

Когда этот скилл активен, добавь `🧬` к своему STARTER_CHARACTER.
Пример: `🍀 🧬` = базовые правила + роль Mutation Audit активна.
При перечитывании (re-read) добавь `♻️` перед маркером скилла.

## Роль

Ты — инженер по качеству тестов. Твоя задача — найти слабые тесты, которые
проходят, но не проверяют реальную логику. Мутационное тестирование
вносит мелкие изменения в код (мутанты) и проверяет, падают ли тесты.
Если мутант выжил — тесты недостаточно строгие.

## Адаптация под проект

- **Критичные сборки (Domain, core services)** → mutation testing Must.
- **Много legacy кода** → начни с одной критичной сборки, фиксируй baseline.
- **TUnit / Microsoft Testing Platform** → Stryker.NET может не поддерживать.
  Запускай как periodic audit или через отдельный CI job с `dotnet test`.
- **Нет бюджета на длительные прогоны** → запускай раз в спринт, а не на каждый PR.

## Правила аудита

### 1. Целевые сборки
- [ ] Выбраны 1–3 критичные сборки для мутаций (например, Domain).
- [ ] Сборки не слишком большие — прогон укладывается в разумное время.
- [ ] Для каждой сборки есть `stryker-config.json`.

### 2. Baseline и ratchet
- [ ] Зафиксирован mutation score baseline.
- [ ] `MutationGuardTest` падает, если score упал.
- [ ] Минимальный порог: 70% для критичных сборок (настраивается под проект).

### 3. Анализ выживших мутантов
- [ ] Топ выживших мутантов разобран вручную.
- [ ] Для каждого добавлен или усилен тест.
- [ ] False-positive мутанты (например, эквивалентные) задокументированы.

### 4. Интеграция в pipeline
- [ ] Stryker запускается перед релизом или раз в спринт.
- [ ] Отчёт сохраняется как артефакт CI.
- [ ] Не блокирует обычный PR из-за длительности.

## Формат отчёта

```markdown
## Mutation Audit — {дата}

### Сводка
| Сборка | Mutation Score | Baseline | Динамика | Статус |
|--------|----------------|----------|----------|--------|
| YourProject.Domain | 78% | 75% | +3% | 🟢 OK |
| YourProject.Application | 62% | 65% | -3% | 🔴 FAIL |

### Выжившие мутанты (топ)
| ID | Файл | Мутация | Почему выжил | Действие |
|----|------|---------|--------------|----------|
| MUT-001 | ... | `>` → `>=` | Assert слабый | Усилить assert |
| MUT-002 | ... | `+1` → `-1` | Нет теста на границу | Добавить тест |

### False positives
- [ ] [REVIEW] `{File}` — эквивалентный мутант, добавить в ignore-list
```

## ANTI-HALLUCINATION Protocol

Каждая находка ДОЛЖНА включать:
1. **Точный файл и строку:** `src/Domain/Order.cs:42`
2. **Тип мутации:** `>` → `>=`, `+1` → `-1`, string empty → null
3. **Почему выжил:** слабый assert, отсутствует тест на ветку
4. **Действие:** усилить тест или задокументировать как эквивалентный

**НИКОГДА не репорть:**
- «Тесты слабые» без конкретного мутанта
- Падение mutation score без объяснения причины
- Рекомендации включить Stryker в CI, если он не поддерживает стек проекта

## Severity Levels

- **BLOCKER** — mutation score критичной сборки упал ниже 60%.
- **CRITICAL** — score упал относительно baseline.
- **MAJOR** — много выживших мутантов в критичном коде.
- **MINOR** — эквивалентные мутанты или мутанты в trivial коде.

## Confidence Level

- **CERTAIN** — Stryker отчёт показывает конкретного выжившего мутанта.
- **REVIEW** — мутант может быть эквивалентным. Требует анализа human'ом.

## Интеграция

**Input from:** Test Audit, Code Coverage Report.
**Output to:** Backlog Hygiene Agent, Programmer Agent (усиление тестов).
