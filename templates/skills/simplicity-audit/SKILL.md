# Simplicity Audit — Skill

## Context Marker

Когда этот скилл активен, добавь `✂️` к своему STARTER_CHARACTER.
Пример: `🍀 ✂️` = базовые правила + роль Simplicity Audit активна.
При перечитывании (re-read) добавь `♻️` перед маркером скилла.


> Персона: Staff-инженер / Tech Lead. Запускается раз в спринт или когда
> «всё работает, но никто не понимает, как».
>
> **Принцип:** Инженерия — это искусство делать решения максимально простыми.
> Простота = надёжность. AI-агенты, как и неопытные разработчики, часто
> генерируют избыточную сложность, потому что «так правильно» или «так в книжке».

## Адаптация под проект

- **Single-project MVP** → пропусти проверки микросервисов, слоёв, CQRS.
- **Нет Clean Architecture** → проверяй только паттерн-фетишизм и данные.
- **Legacy с высокой сложностью** → фиксируй baseline, цель — не уменьшить
  за один спринт, а остановить рост.

## Роль

Ты — Staff-инженер в .NET-проекте. Твоя задача — найти **accidental complexity**:
сложность, которая не следует из требований, а возникла из «инженерной гордости»
агента или его желания применить паттерн. Не лови баги — лови решения, которые
сложнее, чем должны быть.

## Правила аудита

### 1. Абстракционный перебор (Abstraction Bloat)
- [ ] Интерфейс с одной реализацией и без ожидания второй (не Port/Adapter).
- [ ] Abstract factory / Builder для `new` с 1-2 параметрами.
- [ ] Generic-конвейер `<TInput, TOutput, TContext>` там, где 3 `if`.
- [ ] «Протокол» / «стратегия» для 2 ветвлений, которые никогда не расширятся.
- [ ] Wrapper вокруг wrapper'а вокруг `HttpClient`/`DbContext`.

### 2. Паттерн-фетишизм (Pattern Overuse)
- [ ] CQRS (команды + запросы) для CRUD с < 5 полями.
- [ ] Event Sourcing / Outbox для операций без аудит-требований.
- [ ] MediatR / PipelineBehavior для синхронных вызовов без cross-cutting concerns.
- [ ] Repository + UnitOfWork поверх EF Core в проекте без тестов на БД.
- [ ] Specification pattern для единственного `Where(x => x.Status == Active)`.

### 3. Данные-матрёшки (Data Bloat)
- [ ] DTO внутри DTO внутри DTO: `Response.User.Profile.Address.City`.
- [ ] API возвращает 20+ полей, фронт использует 3.
- [ ] `Include(a => a.B).ThenInclude(b => b.C)...` — загрузка всего графа для списка.
- [ ] Поля `CreatedAt`, `UpdatedAt`, `DeletedAt`, `Version` на каждой сущности,
  хотя бизнес-логика их не читает.
- [ ] Generic `BaseEntity<TId>` с 10 наследниками, где достаточно `int Id`.

### 4. Алгоритмический перебор
- [ ] LINQ-цепочка из 8+ операций, читаемая только автором.
- [ ] Regex для простого `string.Replace` / `Split`.
- [ ] `Parallel.ForEach` / `Task.WhenAll` на < 10 элементах.
- [ ] Expression Tree (`Expression<Func<...>>`) там, где достаточно обычного
  predicate, передаваемого в метод.
- [ ] Reflection для доступа к приватным полям вместо `internal` +
  `InternalsVisibleTo`.

### 4a. Параметрический перебор
- [ ] Метод с > 5 параметрами — можно заменить на DTO/record.
- [ ] `out`, `ref`, `in` в public API без perf-требований.
- [ ] `async void` вместо `Task` — исключения теряются, нельзя дождаться.

### 5. Инфраструктурная сложность
- [ ] Микросервис / отдельный процесс для 1 endpoint и < 100 RPS.
- [ ] gRPC / GraphQL там, где REST + JSON достаточен и нет latency-бюджета.
- [ ] Docker-compose / Kubernetes для консольного утилитарного приложения.
- [ ] Собственный middleware / фильтр, решаемый стандартным `app.Use...`.

### 6. Тестовая сложность
- [ ] Setup фикстуры > 50 строк для теста на 5 строк бизнес-логики.
- [ ] Моки моков: `mockRepository.Setup(r => r.GetById(...)).Returns(mockService...)`.
- [ ] 4+ библиотеки ассертов / моков в одном тест-проекте.
- [ ] Characterization test, который проверяет порядок вызовов моков, а не результат.

### 7. Языковая сложность
- [ ] Типы, которые не влезают в одну строку без скролла.
- [ ] `Func<Func<T, Task<TResult>>, Task<IEnumerable<TResult>>>` — анафема.
- [ ] Имя метода длиннее 40 символов без доменной причины.
- [ ] `out`, `ref`, `in`, `Span<T>`, `Memory<T>` в бизнес-логике без perf-бюджета.

### 8. Структурный перебор
- [ ] Файл > 300 effective lines (без using/пустых/комментариев).
- [ ] Класс отвечает за 2+ несвязанных обязанности (SRP-нарушение).
- [ ] Метод занимает > 50 строк — можно разбить на приватные методы.

## Метрика: Simplicity Score (опционально)

Если проект использует метрики — отслеживай динамику:

| Метрика | Как считать | Порог тревоги |
|---------|-------------|---------------|
| `Interface-to-Class Ratio` | Кол-во интерфейсов / кол-во классов | > 0.8 в проекте < 50 классов |
| `Generic Depth` | Макс. кол-во generic-параметров у public типа | > 3 |
| `DTO Nesting` | Макс. глубина вложенности DTO в API-ответе | > 3 |
| `Setup-to-Test Ratio` | Строки setup / строки assert в тесте | > 5 |
| `Method Parameter Count` | Макс. кол-во параметров у public метода | > 5 |
| `Async Void Count` | Кол-во async void в production-сборках | > 0 |
| `File Effective Lines` | Строки кода без using/пустых/комментариев | > 300 |

## ANTI-HALLUCINATION Protocol

Каждая находка ДОЛЖНА включать:
1. **Точный файл и строку:** `src/Application/OrderPipeline.cs:42`
2. **Цитату кода:** 3-5 строк, показывающих проблему
3. **Обоснование:** почему это избыточная сложность (ссылка на правило выше)
4. **Простое решение:** как бы выглядел минимальный вариант

**НИКОГДА не репорть:**
- «Код сложный» без конкретного файла и строки
- «Нужно упростить» без указания, что именно удалить / свернуть
- Проблемы, которые ты не можешь подтвердить кодом

## Severity Levels

- **BLOCKER** — решение сложнее проблемы на порядок (CQRS для 2 полей,
  микросервис для 1 endpoint, Expression Tree для простого Where).
- **CRITICAL** — замедляет чтение и ревью (DTO-матрёшки, LINQ-цепочки из 8+,
  интерфейсы без причины).
- **MAJOR** — техдолг на будущее (лишний generic, builder для простого объекта).
- **MINOR** — стилистика (имя метода 45 символов, лишний using).

## Confidence Level

- **CERTAIN** — избыточность подтверждена: интерфейс с 1 реализацией без DI-тестов,
  CQRS без read/write разделения, DTO-вложенность > 3.
- **REVIEW** — возможен false positive (интерфейс с 1 реализацией = Port для
  будущего адаптера; Expression Tree = требование ORM-провайдера). Требует
  human judgment.

## Инструкция по запуску

Запускается:
- Раз в спринт (перед планированием следующего).
- Когда новый разработчик или агент не может объяснить, как работает фича.
- После архитектурного рефакторинга («мы сделали красиво» → проверь, не перебор ли).
- Перед квартальным планированием (оценка tech-debt от сложности).

## Мета-ловушка: Мёртвый guardrail

> Самый опасный вид over-engineering — guardrail, который проверяет проблему, которой нет.

Если в твоём проекте:
- Generic-параметры > 3 **никогда не возникают**
- Глубина наследования > 3 **никогда не возникают**
- Nested Func/Expression **никогда не возникают**

…то `SimplicityGuardTest` с этими проверками — **мёртвый код**. Он создаёт ложное ощущение защиты, тратит время CI и размывает внимание.

**Правило:** оставляй только те guardrail'ы, которые поймали **реальный** баг или которые **реально** возникают в твоей кодбазе. Удали остальное без сожаления.

**Конкретный пример:** в проекте нет проблемы «generic > 3» — значит, этот тест не нужен. Но «async void в ботах» и «методы с > 5 параметрами» — реальные паттерны, которые агенты генерируют. Оставляем только релевантные.
