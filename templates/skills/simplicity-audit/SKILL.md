# Simplicity Audit — Skill

Optional interaction convention (agent-specific): when this skill is active,
add ✂️ to your STARTER_CHARACTER stack (example: `🍀 ✂️`). Prepend `♻️` when
re-reading the skill. The skill is fully usable without emoji markers.

> **Repo-internal / for methodology archive.** This skill describes a methodological guardrail inside the `dotnet-ai-guardrails` repository. The methodological core (accidental complexity, pattern overuse, data bloat) applies to any project, but the metrics and examples are illustrations, not a universal template.

## Purpose and Non-Goals

Persona: Staff Engineer / Tech Lead. Run once per sprint or when
"everything works, but nobody understands how".

**Principle:** Engineering is the art of making solutions as simple as possible.
Simplicity = reliability. AI agents, like inexperienced developers, often
generate accidental complexity because "that's the right way" or "that's how the book says".

You are a Staff Engineer in a .NET project. Your task is to find **accidental complexity**:
complexity that does not follow from requirements but arises from an agent's
"engineering pride" or desire to apply a pattern. Don't catch bugs — catch solutions
that are more complex than they need to be.

This skill does not judge naming style or enforce a specific architecture —
it flags complexity with no requirements behind it.

## Applicability and Exclusions

- **Single-project MVP** → skip microservice, layer, and CQRS checks.
- **No Clean Architecture** → check only pattern overuse and data bloat.
- **Legacy with high complexity** → establish a baseline; goal is not to reduce
  in one sprint, but to stop growth.

## Required Inputs

- Repository access to application code, tests and project structure.
- Input from: Code Review Agent (pattern observations), Architecture Tests, Tech Debt Agent (stale abstractions).
- Knowledge of actual business requirements behind flagged abstractions (to distinguish essential from accidental complexity).

## Procedure

### 1. Abstraction Bloat
- [ ] Interface with one implementation and no second one expected (not a Port/Adapter).
- [ ] Abstract factory / Builder for `new` with 1-2 parameters.
- [ ] Generic pipeline `<TInput, TOutput, TContext>` where 3 `if` statements suffice.
- [ ] "Protocol" / "Strategy" for 2 branches that will never expand.
- [ ] Wrapper around a wrapper around `HttpClient`/`DbContext`.

### 2. Pattern Overuse
- [ ] CQRS (commands + queries) for CRUD with < 5 fields.
- [ ] Event Sourcing / Outbox for operations without audit requirements.
- [ ] MediatR / PipelineBehavior for synchronous calls without cross-cutting concerns.
- [ ] Repository + UnitOfWork on top of EF Core in a project with no DB tests.
- [ ] Specification pattern for a single `Where(x => x.Status == Active)`.

### 3. Data Bloat (Nested DTOs)
- [ ] DTO inside DTO inside DTO: `Response.User.Profile.Address.City`.
- [ ] API returns 20+ fields, frontend uses 3.
- [ ] `Include(a => a.B).ThenInclude(b => b.C)...` — loading the whole graph for a list.
- [ ] Fields `CreatedAt`, `UpdatedAt`, `DeletedAt`, `Version` on every entity
  when business logic never reads them.
- [ ] Generic `BaseEntity<TId>` with 10 inheritors where `int Id` is enough.

### 4. Algorithmic Overkill
- [ ] LINQ chain of 8+ operations readable only by the author.
- [ ] Regex replaceable by `string.Replace` / `Split`.
- [ ] `Parallel.ForEach` / `Task.WhenAll` on < 10 items.
- [ ] Expression Tree (`Expression<Func<...>>`) where a plain predicate passed to a method is enough.
- [ ] Reflection to access private fields instead of `internal` + `InternalsVisibleTo`.

### 4a. Parameter Bloat
- [ ] Method with > 5 parameters — can be replaced with a DTO/record.
- [ ] `out`, `ref`, `in` in public API without perf requirements.
- [ ] `async void` instead of `Task` — exceptions are lost, cannot be awaited.

### 5. Infrastructure Complexity
- [ ] Microservice / separate process for 1 endpoint and < 100 RPS.
- [ ] gRPC / GraphQL where REST + JSON is sufficient and there is no latency budget.
- [ ] Docker-compose / Kubernetes for a console utility app.
- [ ] Custom middleware / filter solvable by standard `app.Use...`.

### 6. Test Complexity
- [ ] Test fixture setup > 50 lines for a 5-line business logic test.
- [ ] Mocks of mocks: `mockRepository.Setup(r => r.GetById(...)).Returns(mockService...)`.
- [ ] 4+ assertion / mock libraries in a single test project.
- [ ] Characterization test checking mock call order instead of result.

### 7. Language Complexity
- [ ] Types that don't fit in one line without scrolling.
- [ ] `Func<Func<T, Task<TResult>>, Task<IEnumerable<TResult>>>` — unreadable type; split into named intermediate types.
- [ ] Method name longer than 40 characters without domain reason.
- [ ] `out`, `ref`, `in`, `Span<T>`, `Memory<T>` in business logic without a perf budget.

### 8. Structural Bloat
- [ ] File > 300 effective lines (excluding using/blank/comments).
- [ ] Class violates SRP (2+ unrelated responsibilities).
- [ ] Method > 50 lines without compelling reason.

### Metric: Simplicity Score (Optional)

If the project tracks metrics — monitor dynamics:

| Metric | How to Count | Default Threshold (adapt to project baseline) |
|--------|--------------|-----------------|
| `Interface-to-Class Ratio` | # interfaces / # classes | > 0.8 in a project with < 50 classes |
| `Generic Depth` | Max generic parameters on a public type | > 3 |
| `DTO Nesting` | Max DTO nesting depth in an API response | > 3 |
| `Setup-to-Test Ratio` | Setup lines / assert lines in a test | > 5 |
| `Method Parameter Count` | Max parameters on a public method | > 5 |
| `Async Void Count` | async void in production assemblies | > 0 |
| `File Effective Lines` | Lines excluding using/blank/comments | > 300 |

> Thresholds are starting points, not universal rules. Establish the project baseline
> first (current values), then alert on growth beyond baseline, not on absolute numbers.
> Except `Async Void Count` — that one is a defect, not a heuristic.

## Evidence Requirements

Every finding MUST include:
1. **Exact file and line:** `src/Application/OrderPipeline.cs:42`
2. **Code quote:** 3-5 lines showing the problem
3. **Rationale:** why this is excessive complexity (reference rule above)
4. **Simple solution:** what the minimal version would look like

**NEVER report:**
- "Complex code" without a specific file and line
- "Needs simplification" without stating what exactly to remove / collapse
- Issues you cannot confirm with code

## Finding Schema

```text
ID
Severity: BLOCKER | CRITICAL | MAJOR | MINOR
Confidence: CONFIRMED | NEEDS_REVIEW
Category / Control
Evidence: file:line, command output, trace or reproduction
Impact
Recommended action
Owner / disposition
```

## Severity and Confidence

| Severity | Meaning |
|----------|---------|
| **BLOCKER** | Complexity causes defects or blocks changes: nobody can safely modify the code (evidence: reverted PRs, bugs from misunderstood abstraction) |
| **CRITICAL** | Measurably slows reading and review: DTO nesting chains, 8+ LINQ chains, unjustified indirection on a changed-often path |
| **MAJOR** | Future maintenance cost: unnecessary generic, builder for a simple object |
| **MINOR** | Style: long method name, extra using |

| Confidence | Meaning |
|------------|---------|
| **CONFIRMED** | Excess confirmed: interface with 1 implementation and no DI tests, CQRS without read/write separation, DTO nesting > 3 |
| **NEEDS_REVIEW** | Possible false positive (interface with 1 implementation = Port for a future adapter; Expression Tree = ORM provider requirement). Requires human judgment |

Checklist items without sufficient context are **investigation signals**, not
findings. An interface with one implementation, CQRS, or `Task.WhenAll` on a
small collection do not prove a defect by themselves.

## Trigger or Schedule

Run:
- Once per sprint (before planning the next one).
- When a new developer or agent cannot explain how a feature works.
- After architectural refactoring ("we made it beautiful" → check if it's overkill).
- Before quarterly planning (assess tech-debt from complexity).

## Meta-Trap: Unjustified Guardrail

> The most dangerous form of over-engineering is a guardrail that checks for a problem that doesn't exist.

If in your project:
- Generic parameters > 3 **never occur**
- Inheritance depth > 3 **never occurs**
- Nested Func/Expression **never occurs**

...then `SimplicityGuardTest` with these checks is **dead code**. It creates a false sense of security, wastes CI time, and dilutes attention.

**Rule:** a guardrail must be justified by a real incident, a credible threat model, a regulatory requirement, or a documented high-impact failure scenario. Zero triggers alone is not grounds for removal — weigh impact, maintenance cost, false-positive rate, and compensating checks. Removal candidate: low impact + high maintenance cost + existing compensating checks.

## Limitations and Expected False Positives

- Complexity judgments depend on requirements — an abstraction may exist for a planned second consumer. Confirm intent before reporting; mark NEEDS_REVIEW.
- Expected false positives: interface with 1 implementation that is a Port for a future adapter, Expression Tree required by an ORM provider, `Span<T>` inside a proven hot path.
- Metric thresholds (> 0.8 interface ratio, > 300 effective lines) are heuristics, not defects — treat as investigation signals.
- Legacy baselines: pre-existing complexity is tracked for growth, not flagged for immediate removal.
